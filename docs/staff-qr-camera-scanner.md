# Staff QR Camera Scanner

## Status

- Date: 2026-05-19
- Status: Implemented for mobile web browser QA
- Route: `/staffboard/attendance/scan`
- Native mobile app: not required
- File upload/storage changes: not included

No passwords, raw QR payloads, QR tokens, token hashes, session secrets, or real personal data are documented here.

## Supported Flow

1. Staff signs in to JinaCampus.
2. Staff opens StaffBoard Lite > Scan QR.
3. Staff taps Start Camera.
4. Browser requests camera permission.
5. Staff points the camera at the active QR displayed by the school office or gate.
6. The browser decodes the QR payload.
7. The page extracts the raw token in memory only.
8. The existing `scanStaffAttendanceQrAction` submits the token.
9. The existing StaffBoard service validates tenant, branch, authenticated user, staff profile, token expiry, purpose, and duplicate scan rules.
10. The page shows the same safe success/error result card used by manual token entry.

## Manual Fallback

Manual token entry remains available on the same page for:

- denied camera permission
- unsupported browser camera APIs
- unavailable or broken camera hardware
- desktop testing
- low-light or low-end devices where live scanning fails

The fallback accepts the same raw token or `STAFF_ATTENDANCE_QR` JSON payload generated by the QR display page.

## HTTPS Requirement

Camera access requires HTTPS in deployed environments. Localhost works for development.

## Android / iOS QA

Real-device Android Chrome and iOS Safari QA remains recommended before production rollout. Browser camera behavior depends on device permissions, available cameras, OS privacy settings, and the browser media stack.

## Production Scanner Hardening - 2026-07-01

Status: source-level implementation completed; real-device Android/iOS HTTPS QA is still required before claiming QR camera readiness.

Implementation changes:

- The scanner now uses `navigator.mediaDevices.getUserMedia` directly from the Start Camera button and never starts camera access during server rendering or page load.
- Camera support is checked before requesting access, including secure-context, `navigator.mediaDevices`, `getUserMedia`, current origin, user agent, and likely in-app browser detection.
- Camera permission requests use a 12-second timeout so the UI cannot remain stuck on `Requesting camera permission`.
- Rear-camera constraints use `facingMode: { ideal: "environment" }`, with a fallback to `video: true, audio: false` if device constraints fail.
- The preview video is configured for mobile Safari with `autoPlay`, `muted`, `playsInline`, and `webkit-playsinline`, and playback errors are surfaced safely.
- QR decoding now uses `jsQR` on canvas frames from the video preview and uploaded QR image fallback. It does not rely on `BarcodeDetector` or ZXing browser controls.
- Camera tracks and decode loops are stopped on Stop, success, error, `pagehide`, `visibilitychange`, and component unmount.
- The browser submits the raw decoded QR payload string to the server action. The server action parses the payload, extracts the token, and then calls the existing transaction-backed StaffBoard scan service.

Server-side safety remains authoritative:

- Tenant, user, staff profile, branch, role, permission, QR hash, purpose, expiry, duplicate scan state, and audit logging are resolved server-side.
- The client does not submit or control `tenantId`, `branchId`, `staffId`, `actorUserId`, role, permission, attendance status, token hash, or attendance mutation fields.
- Raw QR payloads are not logged, persisted in browser storage, or displayed in result output.

Readiness note:

This hardening removes the known source-level Safari/PWA button-response risk, but it does not certify physical camera readiness. Run approved-HTTPS Android Chrome, iOS Safari, and installed PWA tests for CHECK_IN, CHECK_OUT, invalid QR, denied permission, camera unavailable, duplicate/expired QR, and wrong-branch QR where fixtures exist.
